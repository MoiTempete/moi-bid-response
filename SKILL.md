# 投标技术响应文件编写

## 这个 Skill 做什么

根据用户提供的招标技术规格书（docx），自动完成：

1. **解析招标文件** — 提取章节结构、条款编号、重点标记、**附件格式/响应模板**
2. **生成响应大纲** — 优先按招标自带模板组织，标注核心/制式分类 + 篇幅约束 + 招标条款覆盖映射
3. **逐章编写响应** — 按大纲顺序，采用综合叙述风格，核心优先，每章写完后用户可即时调整
4. **输出 Word 文档** — 汇总为格式化的投标响应文件

## 输入要求

用户需要提供：

- **招标文件 docx 路径**（必需）：技术规格书或招标文件的技术部分
- **项目/公司信息**（可选）：项目名称、投标公司名称，用于文档标题和页眉
- **特殊格式要求**（可选）：如有特定的响应表格格式要求

## 关键概念

### 三种响应组织方式

投标响应文件的组织方式，按优先级依次判断：

**方式 A — 按招标自带模板组织**：
当招标文件包含"附件格式"、"响应文件格式"、"服务方案应包括"等明确的响应模板时，直接按该模板组织。例如：

> 附件格式5：服务方案应包括：(1)实施方案 (2)产品性能 (3)总进度计划及保证措施 (4)质量保障措施 (5)人员配备情况 (6)售后服务保障 (7)合理化建议和优惠条件

**方式 B — 软件工程式组织**（招标无模板时优先采用）：
当招标文件没有规定响应格式时，采用以下标准结构，展示专业性和系统性：

```
1. 项目概述
2. 总体设计原则（5-6条原则，每条50-80字）
3. [可选] 产品生产工艺流程方案（软件工程全流程：需求→设计→编码→测试→部署→运维）
4. 产品功能方案（逐功能模块展开）
5. 技术性能方案（兼容性/响应时间/并发/数据库等）
6. 设计制造标准与安全方案
7. 项目实施与验收方案
8. 售后服务方案
```

此模式参考了真实煤矿安全项目的成功响应文件。

**方式 C — 按招标章节逐章对应**：
当用户明确要求按招标书结构组织，或招标书结构非常清晰且无更好的组织方式时使用。

### 两种合规标注风格

从两批真实投标文件分析中，发现两种有效的响应写法：

**风格 1 — 隐式叙述**（适合企业级/营销型标书）：

不标注"完全响应"，直接以技术叙述方式描述实现方案：

> 平台基于B/S架构构建，支持主流浏览器访问，并兼容PC端与移动端一体化使用。
> 内置统一权限管理体系，提供基于角色的细粒度权限控制，支持多层级组织架构灵活配置。

**风格 2 — 显式引用**（适合工程型/技术型标书）：

每节开头用一句"本章节满足'...'的相关要求"明确对标，然后展开技术叙述：

> 本章节满足"（二）系统功能要求——1、PC端系统功能"中"（1）综合展示：结合
> 煤矿现有的采掘工程平面图，多维展示风险点信息..."的相关要求。
>
> 核心数据总览看板设计：系统登录后默认展示总览看板，涵盖风险点总数...

**选择建议**：
- 招标方是大型国企/集团 → 风格1（隐式叙述，更像企业方案）
- 招标方是技术型单位/煤矿/工厂 → 风格2（显式引用，逐条对标的安心感）
- 用户无偏好时 → 默认风格2（更稳妥，便于评委对标准则）

### 核心编写原则

- 将一个章节内的多条相关要求**融合为连贯的技术叙述**
- **比招标书更具体**：增加具体数字、指标、流程细节
- **核心功能独立展开**：每个关键场景/功能单独成段
- **可选增值章节**：根据项目特点可增加"设计原则"、"生产工艺流程"、"创新性说明"、"应急保障"等

## 工作流

### Step 1 · 解析招标文件

**目标**：提取招标文件的结构、重点内容和响应格式要求。

**操作**：

1. 运行 `python3 ~/.claude/skills/moi-bid-response/scripts/parse_tender.py <招标文件.docx>`
2. 从输出中获取：
   - 章节层级树
   - 重点标记条目列表
   - 完整 JSON 数据

3. **额外检测 — 招标自带响应模板**：
   在解析结果中搜索以下关键词，判断招标是否规定了响应格式：

   - "附件格式"、"响应文件格式"、"服务方案应包括"、"技术方案要求"
   - "实施方案"、"偏离表"、"综合说明"
   - 通常在招标文件的末尾（附录/附件部分）

   如果检测到，提取其要求的章节列表作为大纲骨架。

4. 展示解析结果给用户确认。

**脚本检测的重点标记**：
- 加粗文字（bold）
- 有色/高亮文字
- 特殊符号（★☆※●等）
- 关键词（"重点"/"关键"/"核心"/"必须"/"强制性"/"否决项"/"废标项"）

---

### Step 2 · 风格配置（生成大纲前必须先确认）

**目标**：根据招标文件特征，向用户呈现风格分析和建议，由用户自主选择。

**必须在生成大纲之前完成，不可跳过。**

#### 2a. 分析招标特征

基于 Step 1 的解析结果，提取以下特征用于推荐：

| 特征维度 | 如何判断 | 影响什么 |
|---------|---------|---------|
| 有无附件格式 | 搜索"附件格式"、"响应文件格式"、"偏离表" | 决定组织方式是否有强制模板 |
| 招标方类型 | "集团"、"有限公司"→国企/集团；"煤矿"、"矿井"→技术型单位 | 影响合规标注风格建议 |
| 行业领域 | 关键词匹配（安全/煤矿/冶炼/信息化等） | 影响增值章节选择 |
| 条款密度 | 功能条款数量/总条款比 | 影响粒度建议（细 vs 综合） |
| 标星/加粗比例 | emphasized_count / total_headings | 影响核心/制式分类精度 |

#### 2b. 呈现分析结果并询问用户

展示格式如下（必须用此模板）：

```markdown
## 🔍 招标文件分析

| 特征 | 检测结果 |
|------|---------|
| 附件格式/响应模板 | ✅ 检测到附件格式5 / ❌ 未检测到 |
| 招标方类型 | 国企集团 / 技术型单位 / 政府 / 无法判断 |
| 行业领域 | 煤矿安全 / 冶炼 / 信息化 / ... |
| 条款总数 | X 条，其中标星/加粗 Y 条（占比 Z%） |

---

## 🎨 请选择响应风格

### 一、组织方式

| 选项 | 说明 | 
|------|------|
| **A. 按招标模板** | 以招标附件格式为骨架组织（推荐，因为检测到附件格式） |
| **B. 软件工程式** | 设计原则 → 功能方案 → 性能方案 → 实施 → 售后 |
| **C. 章节对应式** | 按招标技术规格书章节一一对应 |

> 💡 **建议选择 A**，理由：招标文件已明确规定响应格式，按模板组织最稳妥。

### 二、合规标注风格

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| **1. 隐式叙述** | "平台基于B/S架构构建..."，技术口吻自然融入 | 大型国企/集团营销型标书 |
| **2. 显式引用** | 每节"本章节满足'...'的相关要求"后展开 | 技术型招标，便于评委对标准则 |

> 💡 **建议选择 2**，理由：招标方为技术型单位，显式对标更便于评委逐条核对。

---

⚠️ **请回复你的选择**（如"A+2"或"A、隐式叙述"），也可以分别指定。
如需调整建议，直接说明即可。
```

#### 2c. 确认后进入大纲生成

用户确认选择后，将选择记录为：

```
组织方式: [A/B/C]
合规风格: [1/2]
```

然后在 Step 3 中按此配置生成大纲和正文。

### Step 3 · 生成响应大纲（关键步骤，必须等用户确认）

**目标**：基于用户选择的风格配置，生成带分类标注、篇幅约束和招标条款覆盖映射的响应大纲。

#### 3a. 按选定的组织方式生成大纲骨架

- **方式 A**：以招标模板章节为一级骨架，必要时根据技术规格书补充细化
- **方式 B**：采用标准结构 `项目概述 → 设计原则 → [可选]生产工艺流程 → 产品功能方案 → 技术性能方案 → 标准与安全方案 → 实施与验收 → 售后服务`
- **方式 C**：按招标技术章节一一对应

#### 3b. 内容分类

| 分类 | 判断依据 |
|------|---------|
| `[核心·标星]` | 招标文件中有加粗/标星/标色/关键词标记的条款 |
| `[核心]` | 技术方案/系统架构/功能指标/建设内容等实质性内容，但无特殊格式标记 |
| `[制式·重点]` | 售后/培训/维保/质保等通用性内容，但招标有特殊要求 |
| `[制式]` | 无特殊标记的通用标准条款（验收、文档、项目管理等） |

对每个分类决策，附一句理由。

#### 3c. 篇幅约束

| 分类 | 建议字数 | 说明 |
|------|---------|------|
| `[核心·标星]` | 1500-3000 字 | 每个子功能/场景独立展开，加具体指标 |
| `[核心]` | 800-1500 字 | 综合叙述，覆盖多个相关条款 |
| `[制式·重点]` | 400-800 字 | 标准化展开，适当细化 |
| `[制式]` | 200-500 字 | 标准模板，简洁清晰 |

#### 3d. 覆盖映射

在大纲中增加一列，标注每个响应章节对应覆盖了哪些招标条款，确保**无遗漏**。

#### 3e. 输出格式

用户已选风格应在标题区体现：

```markdown
## 📋 投标技术响应大纲

**项目**：<项目名>
**招标文件**：<文件名>
**组织方式**：方式 A / B / C
**合规风格**：风格 1（隐式叙述）/ 风格 2（显式引用）
**生成时间**：<时间>

---

### 📌 核心章节（共 X 章，预估总字数 Y）

| # | 响应章节 | 覆盖招标条款 | 分类 | 篇幅 | 理由 |
|---|---------|-------------|------|------|------|
| 1 | 软件平台架构设计 | 五.1, 四.2.2 | [核心·标星] | ~1500字 | 招标加粗"工业智能体平台"，覆盖B/S、权限、工作流等7条技术指标 |
| 2 | AI算法模型集成方案 | 五.2.1-2.7 | [核心·标星] | ~2500字 | 7类AI场景为核心建设内容，每场景独立展开 |
| ... | ... | ... | ... | ... | ... |

### 📋 一般制式（共 X 章，预估总字数 Y）

| # | 响应章节 | 覆盖招标条款 | 分类 | 篇幅 | 理由 |
|---|---------|-------------|------|------|------|
| 7 | 售后服务保障 | 七.3, 一.5 | [制式·重点] | ~600字 | 质保5年+2小时响应，有特殊要求 |
| 8 | 人员培训方案 | 七.2 | [制式] | ~400字 | 标准培训条款 |
| ... | ... | ... | ... | ... | ... |

### 📊 汇总

| 指标 | 数值 |
|------|------|
| 核心章节 | X 章 |
| 制式章节 | Y 章 |
| 预估总字数 | Z 字 |
| 招标条款覆盖率 | 100%（确认无遗漏） |
| 建议编写顺序 | 核心 → 制式 |

---

⚠️ **请审核以上大纲，你可以：**
- 调整任意章节的分类或篇幅
- 增删、合并、拆分章节
- 调整编写顺序
- 修改覆盖映射

确认后我将开始逐章编写。
```

#### 3f. 等待用户确认

- **必须等待用户明确确认**才能进入 Step 4
- 用户可做任何调整
- 如果用户只回复"确认"/"OK"/"开始"等肯定词，直接进入 Step 3

### Step 4 · 逐章编写响应

**目标**：按用户选择的大纲和合规风格，逐章编写技术响应。

**编写顺序**：先写所有 `[核心]` 类别章节，再写 `[制式]` 类别章节。

#### 编写原则（重要）

基于对真实投标响应文件的分析，遵循以下编写风格。**具体执行取决于用户在 Step 2 中选择的风格：**

**通用原则（两种风格均适用）**：

1. **比招标书更具体**：
   - 招标说"支持高并发" → 响应写"支持1000+用户同时在线，每秒200+并发请求"
   - 招标说"AI模型" → 响应写"检测精度≥98%，误报率≤2%，响应时间≤2秒"
   - 能加数字的地方加数字，能列举的地方列举

2. **核心功能独立展开**：
   - 每个关键功能/场景各自成段，有独立的标题和指标
   - 如：7个AI场景每个都有独立的准确率、覆盖范围、响应时间

3. **制式内容标准化但完整**：
   - 售后服务/培训等章节按标准模板展开
   - 仍然要具体（故障分级、响应时效、人员安排等）
   - 使用编号列表增强可读性和专业感

4. **不确定处留标记**：
   - 无法确认的技术细节标注 `【待补充】`
   - 需要用户提供的公司/产品信息标注 `【待填写】`

**风格 1（隐式叙述）特有原则**：

5. **综合叙述，不逐条标注**：
   - 将同一章节内相关要求融合为连贯的技术叙述
   - 不说"完全满足"，而是直接描述如何实现
   - 用"我方"、"本项目"、"系统"为主语做正面描述

**风格 2（显式引用）特有原则**：

6. **逐节对标，先引用后展开**：
   - 每节（模块设计）开头用一句"本章节满足'（X）XXX'中'（X）XXX：...'的相关要求。"明确对标
   - 引文必须引用招标原文完整内容（用引号包裹）
   - 然后再展开技术叙述
   - 避免写成逐条的"要求→响应"格式，而是在引用后以模块为单位综合叙述

#### 每章编写流程

1. **回顾招标原文**：从 Step 1 解析结果中找到该响应章节覆盖的所有招标原条款，仔细阅读
2. **查阅编写规范**：参考 `references/writing-guide.md` 中的编写规范和篇幅指南
3. **生成响应内容**：
   - 按上述综合叙述风格撰写
   - 控制篇幅在设定范围内（±20%）
   - 确保覆盖映射中标注的所有招标条款
4. **自检覆盖度**：确认该章所覆盖的招标条款均已纳入响应
5. **展示草稿**：以清晰的 Markdown 格式输出
6. **等待用户反馈**：用户可确认、修改、重写，处理完后进入下一章

### Step 5 · 汇总输出 Word 文档

**目标**：将所有确认的响应内容生成为结构化的 docx 技术文档。

#### 输出文档格式（匹配往期2真实格式 + 国标多级自动编号）

| 元素 | 格式 |
|------|------|
| 页面 | A4 (21cm × 29.7cm)，上/下边距 2.5cm，左/右边距 3.2cm |
| 文档标题 (H1) | 宋体加粗，~16pt，居中，行距 1.5，不参与编号 |
| 一级标题 (H2) | 黑体，~15pt，行距 1.5，段前段后 12pt |
| 二级标题 (H3) | 黑体，~14pt，行距 1.5，段前段后 12pt |
| 三级标题 (H4) | 黑体，~14pt，行距 1.5 |
| 四级标题 (H5) | 黑体，~14pt，行距 1.5 |
| 正文段落 | 宋体，12pt，行距 1.5 倍 |
| 表格正文 | 宋体，~11pt |
| 表格表头 | 黑体，~11pt，加粗，灰底 |
| 页眉页脚 | 无需设置（简洁风格） |

#### 国标多级自动编号方案

使用 Word 多级列表机制，标题编号完全自动生成，**JSON 中只需填写标题文字，禁止手写编号**。

| 标题层级 | 编号格式 | 示例 | Word numFmt |
|---------|---------|------|-------------|
| H1（文档标题） | 无编号 | 技术响应方案 | — |
| H2（一级目录） | `一、` `二、` | 一、项目概述 | chineseCountingThousand |
| H3（二级目录） | `1. ` `2. ` | 1. 建设背景 | decimal |
| H4（三级目录） | `1.1. ` `1.2. ` | 1.1. 总体架构 | decimal |
| H5（四级目录） | `1.1.1. ` | 1.1.1. 数据库设计 | decimal |
| H6（五级目录） | `①` `②` | ①索引优化 | （注） |
| H7（六级目录） | `(1)` `(2)` | (1)创建复合索引 | （注） |

> **注**：H6（①）和 H7（(1)）在 Word 多级列表中实现方式受限，建议仅在确实需要时使用，或在编写时优先将内容控制在 H2-H5 层级内。如必须使用，由脚本尽力支持。

```

  1. 项目建设背景               ← H3
```

**JSON 规则**：
- `title`：文档标题（H1，不编号）
- `level`：标题层级（2-7），对应 H2-H7
- `content` 三种元素类型：
  - `{"type": "p", "text": "..."}` — 正文段落
  - `{"heading": "...", "level": N, "content": [...]}` — 嵌套子章节
  - `{"type": "table", "headers": [...], "rows": [[...], ...]}` — 嵌入式表格
- **标题文字中禁止手写编号**，所有编号由脚本自动生成

2. **运行生成脚本**：

```bash

```

支持两种模式：stdin（兼容旧用法）和文件模式（推荐，避免管道编码问题）：
```bash
python3 ~/.claude/skills/moi-bid-response/scripts/generate_docx.py output.docx /path/to/data.json
```

3. **大文档分批生成与合并策略**：当响应总字数超过 15,000 字或 JSON 体积超过 50KB 时，单次传输可能受限。采用以下策略确保内容完整不失真：
   - **分批生成**：将各章（或各节）分别生成为独立 .docx 文件，每份文件仅含 1-2 章
   - **合并脚本**：使用 `scripts/merge_docx.py` 将各章 docx 按顺序合并为完整文档
   - **直接生成**：提供一个可在本地执行的独立 Python 脚本（用 `python-docx` 绕过 JSON 中间格式），用户本地运行一步生成完整文档
   - **内容保真**：无论采用哪种方式，已确认的 Step 4 原文段落必须 1:1 保留，禁止以"适配格式"为由合并或删减段落

4. 告知用户文件路径，提醒可后续手动插入架构图、部署图等图片。

## 关键规则

1. **风格必须先确认**：Step 2 必须先完成风格配置（组织方式 + 合规风格），用户确认后才能进入 Step 3 大纲生成
2. **大纲必须等用户确认**：Step 3 生成大纲后不可跳过，用户确认后才能进入 Step 4 编写
3. **逐章确认**：每章写完后展示给用户，不要一次性写完
4. **核心优先**：先写核心章节，再写制式章节
5. **依风格执行**：风格1用隐式叙述，风格2用显式引用，不混用
6. **比招标更具体**：增加数字、指标、流程细节
7. **篇幅约束**：控制在设定范围内，偏差不超过 ±20%
8. **不要编造**：不确定处标注 `【待补充】`
9. **确保全覆盖**：每条招标要求都应在响应中找到对应
10. **Step 4 内容不可逆**：逐章编写经用户确认后的内容即视为终稿。Step 5 生成文档时，**严禁**对已确认的内容做任何精简、压缩、合并段落或删改。若因工具传输限制导致无法一次性生成完整文档，须采用分批生成后合并的方式，确保已确认的每一个字都进入最终文档。生成后须进行字数自检，与 Step 4 确认字数对比，差距超过 5% 须主动说明原因并补救
